# Import Process

Use the following process to import data into the Platform:
1. Format the relevant importer
2. Generate a bearer token
3. Import your data
4. Evaluate responses

## Note on best practices

During the import process, the Company recommends adhering to the following best practices:

| Topic              | Description                                                                                                                                                                                            |
|--------------------|--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| **Import thresholds** | To maximize the speed at which your data is processed, it is highly recommended that you separate your Product and Price records into multiple batches/requests, e.g., importing batches of 100 products, 10 requests at a time, and a 1 second pause between each request. |
| **Manual imports** | The process described on this page must be followed when manually importing data into the Platform. You can use this process for establishing an initial master dataset, running tests against sample populations, or troubleshooting backend automation issues. |
| **Automated imports** | After completing the initial Master Data integration process, it is recommended that all future data imports are scheduled to run at pre-configured intervals. This automated setup ensures critical data remains up-to-date and aligned with your systems of record. |

---

## 1. Format the relevant importer

To get started on the Platform, you must first understand how your underlying master data will be used, how each data type is intertwined, and why this data is necessary for performing basic functions on the POS application.

### Data format

Each set of master data is typically derived from an external system of record, e.g., from your ERP Systems. To upload your master data into the Platform, you must use a set of dedicated Importers, i.e., schemas that enable you to easily convert your ERP data to JavaScript Object Notation (JSON) format.

This JSON formatting ensures:
- You can bypass custom schema development and immediately start the import process.
- Your imports are mapped correctly and processed successfully.
- Your initial data payload can meet the requirements of any custom UI or legacy system.

### Import requirements

The Company provides five dedicated Importers for each type of master data. When using bulk importers for the first time, you must map your upstream data to equivalent required/optional fields contained in each Importer. To successfully integrate to the Company's Digital Platform, you must begin by importing all five master data types in the order listed below.

| Order         | Data type          |
|---------------|--------------------|
| First import  | Locations          |
| Second import | Merchandise Groups |
| Third import  | Products           |
| Fourth import | Prices             |
| Fifth import  | Taxes              |

## 2. Generate a bearer token

Bearer tokens use an access key (i.e., a distinct text string) to enable authentication requests. You will use these tokens to access the Company’s REST API importer endpoints.

To generate a bearer token, use the required variables, request structure, and request process outlined below.

### Required variables

Each of the following variables are required in order to successfully process your request:

| Name                        | Description                                                                                                                                  |
|-----------------------------|----------------------------------------------------------------------------------------------------------------------------------------------|
| `$EnvironmentType`          | The name and type of environment for which you are requesting access, e.g., `dev-env`.                                                       |
| `$CustomerCode`             | The short-form customer identifier associated to the environment for which you are requesting access, e.g., `acme`. (Note: This code is unique to all customers and is usually 2-3 letters in length). |
| `$Region`                   | The relevant cloud provider region, e.g., `us-east-1`. (Note: To access this information, log into your password manager and search for the "Cloud Account - Customer Info" record). |
| `$Base64Auth`               | A string authenticated for HTTP Basic Auth that contains `$ClientId` and `$ClientSecret` values. This string is encoded as follows: `Authorization: Basic <$Base64Auth($ClientId:$ClientSecret)>` |
| `$GrantType`                | A body parameter used to define your OAuth flow, e.g., `client_credentials`.                                                                 |
| `$ClientId` / `$ClientSecret` | Unique, alphanumeric public and private client identifiers that are specific to each `$EnvironmentType`. (Note: To access this information, log into your password manager and search for the "Cloud Cognito Client - Integrations" record). |
| `$ResourceServerIdentifier` | A unique identifier for the resource server, e.g., `acme-dev-env-integrations`. (Note: To access this information, log into your password manager and search for the "Cloud Cognito Client - Integrations" record). |
| `$ScopeName`                | The scope name, e.g., `integrations`. (Note: To access this information, log into your password manager and search for the "Cloud Cognito Client - Integrations" record). |

### Request structure

Use the command structure below to generate a bearer token for your respective environment.

```bash
curl --X POST \
  https://$EnvironmentType-$CustomerCode.auth.$[Region.amazoncognito.com/oauth2/token](https://Region.amazoncognito.com/oauth2/token) \
  -H 'Authorization:Basic $Base64Auth' \
  -H 'content-type: application/x-www-form-urlencoded' \
  -d 'grant_type=$GrantType&client_id=$ClientId&scope=$ResourceServerIdentifier/$ScopeName'
