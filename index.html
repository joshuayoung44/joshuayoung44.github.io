                        <h5>Step 1: Format the Relevant Importer</h5>

                        <p>Each set of master data is typically derived from an external system of record, e.g., from your ERP Systems. To upload your master data into ACME, you must use a set of dedicated ACME Importers, i.e., schemas that enable you to easily convert your ERP data to JavaScript Object Notation (JSON) format.</p>

                        <p>To successfully integrate to the ACME Digital Platform, you must begin by importing all five master data types in the order listed below.</p>

                        <ol>

                            <li>Locations</li>

                            <li>Merchandise Groups</li>

                            <li>Products</li>

                            <li>Prices</li>

                            <li>Taxes</li>

                        </ol>


                        <h5>Step 2: Generate a Bearer Token</h5>

                        <p>Bearer tokens use an access key (i.e., a distinct text string) to enable authentication requests. You will use these tokens to access ACME's REST API importer endpoints.</p>

                        

                        <h5>Step 3: Import Your Data</h5>

                        <p>After formatting the relevant importer and generating a bearer token, you can now import your data via a REST API request. This request must contain the following components:</p>

                        <ul>

                            <li><strong>Method:</strong> POST</li>

                            <li><strong>URL:</strong> The relevant REST API endpoint</li>

                            <li><strong>Authorization Type:</strong> Bearer Token</li>

                            <li><strong>Header:</strong> `content-type: application/JSON`</li>

                            <li><strong>Body:</strong> Your data mapped to the relevant ACME Importer.</li>

                        </ul>


                        <h5>Step 4: Evaluate Responses</h5>

                        <p>After importing data into ACME, two forms of responses may be returned: API responses and SQS queue responses. When your import is processed, one of the API responses listed below will be returned.</p>


                        <h4>Sample 2: Product Importer API Reference</h4>

                        

                        <h5>Overview</h5>

                        <p>The ACME Products Importer provides the necessary structure for mapping and organizing all relevant product records and characteristics, e.g., SKUs, item names, and attributes.</p>

                        <p>Though prices may differ based on store location, product characteristics typically remain static. As a result, product records are often imported at the root-level of a location hierarchy, making them available to your entire store network.</p>


                        <h5>Example Use Cases</h5>

                        <ul>

                            <li>The ability to make products available to the POS and proceed to sales.</li>

                            <li>The ability to automatically replicate product records to POS devices for use in offline mode.</li>

                            <li>The option to manually replicate product records to POS devices as needed.</li>

                            <li>The ability to display product names and descriptions in a preferred language (at one or many store locations).</li>

                            <li>The ability to push actions to the POS based on values defined in product-level properties.</li>

                        </ul>


                        <h5>Integration Notes</h5>

                        <p>Products are maintained in each customer's ERP system. Therefore, to interact with your product inventory via the ACME application, you must upload specific data elements from your ERP system into the ACME platform.</p>

                        <p>However, ERP systems do not typically contain all of the information required to successfully map your product data. This is due to the fact that the Products Importer contains a set of ACME-specific properties. As a result, you will likely need to use an API management tool to package your API request.</p>

                        <h3>POS Cart User Guide</h3>

                        <blockquote>This guide explains the features and functions of a Point of Sale (POS) Cart screen, written for a non-technical audience like retail store associates. It focuses on clear, step-by-step instructions for completing daily tasks.</blockquote>

                        

                        <h4>Introduction</h4>

                        <p>The Cart operates as the POS application's idle screen. You can access it from each of the following locations:</p>

                        <ul>

                            <li>After logging into the POS UI, you will land on the Cart screen by default.</li>

                            <li>After completing a transaction, you will be automatically redirected to an empty Cart screen.</li>

                            <li>On the upper-left side of the POS UI, expand the main menu and click Cart.</li>

                        </ul>

                        <p>At a high level, the POS Cart is separated into two sections:</p>

                        <ul>

                            <li><strong>Item cards:</strong> Contains information related to each transaction item, e.g., the Image, SKU, Quantity, Unit Price, and Total Price.</li>

                            <li><strong>Checkout Summary panel:</strong> Contains transaction-level information, e.g., the Subtotal, applied Promotion and Discount values, Taxes, and Transaction Total.</li>

                        </ul>

                        <h3>Change Notice Process Definition</h3>

                        <blockquote>This document defines the internal process, criteria, and scope for creating and delivering customer-facing Change Notices. This ensures clear and timely communication about product updates for stakeholders and is crucial for maintaining trust and alignment.</blockquote>

                        

                        <h4>Criteria and Scope</h4>

                        <p>Change Notices are brief, release-specific documents that contain timely use case instructions, best practices, and/or implementation requirements for specified customer audiences.</p>

                        <p>While release notes describe the highest level changes to the product, Change Notices typically go into greater detail on how and why a given change impacts a particular user type.</p>

                        

                        <h4>Template Structure</h4>

                        <p>Change Notices are broken into the following major sections:</p>

                        <ul>

                            <li><strong>Who does this impact?</strong> Call out which customer types are impacted by the change.</li>

                            <li><strong>Effective date:</strong> Provide the version number containing the change.</li>

                            <li><strong>What is changing?</strong> Describe the customer-facing elements of the application that changed.</li>

                            <li><strong>Why is the change being made?</strong> Provide brief context around why the update was implemented.</li>

                            <li><strong>What impact will it have on customers?</strong> Describe what the impacted customer type must do as a result of the change.</li>

                        </ul>
