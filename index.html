Note on best practices

    
During the import process, Our Platform recommends adhering to the following best practices:


    

        
            

                Topic
                Description
            

        
        
            

                Import thresholds
                To maximize the speed at which your data is processed, it is highly recommended that you separate your Product and Price records into multiple batches / requests, e.g., importing batches of 100 products, 10 requests at a time, and a 1 second pause between each request.
            

            

                Manual imports
                The process described on this page must be followed when manually importing data into Our Platform. You can use this process for establishing an initial master dataset, running tests against sample populations, or troubleshooting backend automation issues.
            

            

                Automated imports
                After completing the initial Master Data integration process, it is recommended that all future data imports are scheduled to run at pre-configured intervals. This automated setup ensures critical data remains up-to-date and aligned with your systems of record.
            

        
    


    
1. Format the relevant importer

    
To get started on the Our Platform, you must first understand how your underlying master data will be used, how each data type is intertwined, and why this data is necessary for performing basic functions on the POS application.


    
Data format

    
Each set of master data is typically derived from an external system of record, e.g., from your ERP Systems.

    
To upload your master data into Our Platform, you must use a set of dedicated Our Platform Importers, i.e., schemas that enable you to easily convert your ERP data to JavaScript Object Notation (JSON) format.

    
This JSON formatting ensures:

    
    
        
  • You can bypass custom schema development and immediately start the import process.
    • 
        
  • Your imports are mapped correctly and processed successfully.
    • 
        
  • Your initial data payload can meet the requirements of any custom Ul or legacy system.
    • 
    


    
Import requirements

    
Our Platform provides five dedicated Importers for each type of master data.

    
When using bulk importers for the first time, you must map your upstream data to equivalent required / optional fields contained in each Our Platform Importer.

    
To successfully integrate to the Our Digital Platform, you must begin by importing all five master data types in the order listed below.


    

        
            

                Order
                Data type
            

        
        
            

                First import
                Locations
            

            

                Second import
                Merchandise Groups
            

            

                Third import
                Products
            

            

                Fourth import
                Prices
            

            

                Fifth import
                Taxes
            

        
    


    
2. Generate a bearer token

    
Bearer tokens use an access key (i.e., a distinct text string) to enable authentication requests.

    
You will use these tokens to access Our Platform's REST API importer endpoints.

    
To generate a bearer token, use the required variables, request structure, and request process outlined below.


    
Required variables

    
Each of the following variables are required in order to successfully process your request:

    

        
            

                Name
                Description
            

        
        
            

                $EnvironmentType
$CustomerCode		
                The name and type of environment for which you are requesting access, e.g., sample-dev
The short-form customer identifier associated to the environment for which you are requesting access, e.g., abc.
Note: This code is unique to all Our Platform customers and is usually 2-3 letters in length.		
            

            

                $Region
                The relevant cloud provider region, e.g., us-west-2.
Note: To access this information, log into a password manager and search for the "Cloud Account - Customer Info" record.		
            

            

                $Base64Auth
$GrantType		
                A string authenticated for HTTP Basic Auth that contains $ClientId and $ClientSecret values. This string is encoded as follows:
Authorization: Basic <$Base64Auth($ClientId:$ClientSecret)>
A body parameter used to define your OAuth flow, e.g., client_credentials		
            

            

                $ClientId / $ClientSecret
                Unique, alphanumeric public and private client identifiers that are specific to each $EnvironmentType.
Note: To access this information, log into a password manager and search for the "Cloud Cognito Client - Integrations" record.		
            

            

                $ResourceServerIdentifier
                A unique identifer for the resource server, e.g., abc-sample-dev-integrations
Note: To access this information, log into a password manager and search for the "Cloud Cognito Client - Integrations" record.		
            

            

                $ScopeName
                The scope name, e.g., integrations.
Note: To access this information, log into a password manager and search for the "Cloud Cognito Client - Integrations" record.		
            

        
    


    
Request structure

    
Use the command structure below to generate a bearer token for your respective environment.

    

    curl -X POST https://$EnvironmentType-$CustomerCode-auth.$Region.cloudprovider.com/oauth2/token -H 'Authorization: Basic $Base64Auth' -H 'content-type: application/x-www-form-urlencoded' -d 'grant_type=$GrantType&client_id=$ClientId&scope=$ResourceServerIdentifier/$ScopeName'
    


    
Example request

    
Each variable (e.g., $EnvironmentType) will be replaced with values specific to your request.

    
For example, the following request contains the necessary information to access abc-sample-dev-integrations:

    

    curl -X POST https://sample-dev-abc-auth.us-west-2.cloudprovider.com/oauth2/token -H 'Authorization: Basic [REDACTED_BASE64_STRING]' -H 'content-type: application/x-www-form-urlencoded' -d 'grant_type=client_credentials&client_id=[REDACTED_CLIENT_ID]&scope=abc-sample-dev-integrations/integrations'
    


    
Request process

    
The instructions below assume you will submit your request via the command line. If you are unfamiliar with the command line, you can generate tokens using a REST client.


    
Generate a Basic Auth token

    
In your terminal, enter the following command:

    

    echo -n "$ClientId:$ClientSecret" | base64
    

    
Once your token is returned, populate the Authorization: Basic $Base64Auth component of your request.


    
Generate a bearer token

    
In your terminal, submit your formatted curl -X POST request. Once successfully processed, you will receive a response in the following format:

    

    {"access_token": "xxxxxXXXXXXXXXX", "expires_in": 3600,"token_type": "Bearer"}
    

    
Note: Once generated, this "access_token":"xxxxxxxxxxxxxxx" can be used for Bearer Token Authentication (instead of Basic Authentication) when requesting access to the desired Our Platform endpoint in a REST client.


    
3. Import your data

    
After formatting the relevant importer and generating a bearer token, you can now import your data via a REST API request.


    
Request components

    
This request must contain the following components:

    

        
            

                Component(s)
                Value(s)
            

        
        
            

                Method
                POST
            

            

                URL
                The relevant REST API endpoint
            

            

                Authorization type
                Bearer Token
            

            

                Username and password
                Provided by your customer success lead.
            

            

                Header
                A single header with the following key/value pair: content-type (key) and application/JSON (value)
            

            

                Body
                Your data mapped to the relevant Our Platform Importer.
            

        
    

    
Note: When testing API calls manually, it is recommended that you use REST API software.


    
Endpoint components

    
Each importer endpoint is formatted as follows:

    

    https://{{gatewayURL}}/importers/<importer-name>
    


    

        
            

                Component
                Description
            

        
        
            

                Gateway URL
                The URL used to log into the application on a device—including a Hypertext Transfer Protocol Secure (i.e., https:// ) prefix.
            

            

                Data entity name
                The name of the data entity to be imported—including an "importers/" prefix (e.g., importers/prices ).

                    "ProductCode": "PROD001",
                    "SKU": "PROD001SKU",

                    },

                        "f": 1,

                    },
                    "CurrencyId": "USD",

                }
            ]
        }
    }'
    


    
4. Evaluate responses

    
After importing data into Our Platform, two forms of responses may be returned:

    
API responses

    
When your import is processed, one of the API responses listed below will be returned.

    

        
            

                Status code
                Message
                Description
            

        
        
            

                200
                requestId
                A generic response indicating the import was received. Note: Receiving a 200 response does not necessarily indicate your request was successfully processed. For a detailed description of your request status, use the requestId to locate and review the response delivered to your SQS queue. Currently, 204 (partial success) and 207 (complete success) status codes are not returned.
            

            

                400
                bad request
                Indicates the request cannot be fulfilled due to bad syntax.
            

            

                401
                unauthorized
                Indicates an authentication failure, i.e., incorrect or missing credentials were used in the request.
            

            

                403
                forbidden
                Indicates the endpoint is incorrect or incorrectly formatted.
            

            

                429
                too many requests
                Indicates too many requests were submitted at once, and the application failed to process the data.
            

            

                500
                internal server error
                Indicates the connection cannot be made, and the system or import service may be down. When this occurs, contact your administrator. Note: If stageVariables are unavailable, the API will respond with a 500 error for all imports.
            

            

                502
                bad gateway
                Indicates the API URL was formatted incorrectly.
            

            

                504
                lambda timeout
                Indicates the file is too large, and the request timed out before it was fully processed.
            

        
    


    
SQS queue responses

    
In addition to the potential API error responses, one of the messages below will also be available in your associated SQS queue.

    
When viewing these messages, note:

    
    
        
  • Each SQS message explains the import type (e.g., Product, Price, MMG, etc.) and status (e.g., success / failure).
    • 
        
  • You can use the unique requestId (contained in your API response) to match each SQS ID/Body to your attempted import.
    • 
    


    

        
            

                Message type
                Description
            

        
        
            

                Successful Import
                Indicates the import was processed successfully.
            

            

                Schema validation error
                Indicates the import is missing a required field. Note: If there are multiple schema validation errors, the SQS import message displays the instance number(s) corresponding to the rejected JSON schema(s), e.g., instance.Taxes[50].
            

            

                Property / field validation error
                Indicates the import contains invalid values (e.g., inputting an EndDate that precedes a StartDate, inputting a location that doesn't exist in the database, etc.). Note: If there are multiple schema validation errors, the SQS import error queue displays each rejected JSON schema ID separated by a comma.
            

            

                Failure to post to Database
                Indicates the imported JSON schema contains updates to a field not contained in the database. Note: When you receive this response, the SQS import error queue displays the rejected JSON schema ID, as well as the reason, e.g., Document update conflict.
            

            

                Too many requests
                Indicates you have submitted too many requests and exceeded your rate limit allowance.

  • After an attempted Product, Prices, Linked Products, or Product Attributes import, this error message includes the relevant productId with a link to the S3 bucket containing the corresponding request payload.
