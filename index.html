Sample 1: Master Data Import Process

Overview

Use the following process to import data into the ACME Platform:

  1. Format the relevant importer
  2. Generate a bearer token
  3. Import your data
  4. Evaluate responses
Best Practices
Topic Description
Import Thresholds To maximize the speed at which your data is processed, it is highly recommended that you separate your Product and Price records into multiple batches / requests, e.g., importing batches of 100 products, 10 requests at a time, and a 1 second pause between each request.
Manual Imports The process described on this page must be followed when manually importing data. You can use this process for establishing an initial master dataset, running tests against sample populations, or troubleshooting backend automation issues.
Automated Imports After completing the initial Master Data integration process, it is recommended that all future data imports are scheduled to run at pre-configured intervals. This automated setup ensures critical data remains up-to-date and aligned with your systems of record.
Step 1: Format the Relevant Importer

Each set of master data is typically derived from an external system of record, e.g., from your ERP Systems. To upload your master data into ACME, you must use a set of dedicated ACME Importers, i.e., schemas that enable you to easily convert your ERP data to JavaScript Object Notation (JSON) format.

To successfully integrate to the ACME Digital Platform, you must begin by importing all five master data types in the order listed below.

  1. Locations
  2. Merchandise Groups
  3. Products
  4. Prices
  5. Taxes
Step 2: Generate a Bearer Token

Bearer tokens use an access key (i.e., a distinct text string) to enable authentication requests. You will use these tokens to access ACME's REST API importer endpoints.

Step 3: Import Your Data

After formatting the relevant importer and generating a bearer token, you can now import your data via a REST API request. This request must contain the following components:

  • Method: POST
  • URL: The relevant REST API endpoint
  • Authorization Type: Bearer Token
  • Header: `content-type: application/JSON`
  • Body: Your data mapped to the relevant ACME Importer.
Step 4: Evaluate Responses

After importing data into ACME, two forms of responses may be returned: API responses and SQS queue responses. When your import is processed, one of the API responses listed below will be returned.

Sample 2: Product Importer API Reference

Overview

The ACME Products Importer provides the necessary structure for mapping and organizing all relevant product records and characteristics, e.g., SKUs, item names, and attributes.

Though prices may differ based on store location, product characteristics typically remain static. As a result, product records are often imported at the root-level of a location hierarchy, making them available to your entire store network.

Example Use Cases
  • The ability to make products available to the POS and proceed to sales.
  • The ability to automatically replicate product records to POS devices for use in offline mode.
  • The option to manually replicate product records to POS devices as needed.
  • The ability to display product names and descriptions in a preferred language (at one or many store locations).
  • The ability to push actions to the POS based on values defined in product-level properties.
Integration Notes

Products are maintained in each customer's ERP system. Therefore, to interact with your product inventory via the ACME application, you must upload specific data elements from your ERP system into the ACME platform.

However, ERP systems do not typically contain all of the information required to successfully map your product data. This is due to the fact that the Products Importer contains a set of ACME-specific properties. As a result, you will likely need to use an API management tool to package your API request.

Example Payload: Product with Variants
{
  "Products": {
    "locationId": "acme_global",
    "Product": [{
      "id": "999999",
      "RetailerId": "acme_global",
      "ProductCode": "999999",
      "Name": "Sample product variant name",
      "ProductType": "StyleColorSize",
      "ProductSkus": {
        "Sku": [{
          "id": "999999-BK-02",
          "Name": "Sample product variant name",
          "Variants": {
            "type": [{"id": "color", "value": "black"}, {"id": "size", "value": "02"}]
          }
        }, {
          "id": "999999-WH-04",
          "Name": "Sample product variant name",
          "Variants": {
            "type": [{"id": "color", "value": "white"}, {"id": "size", "value": "04"}]
          }
        }]
      }
    }]
  }

POS Cart User Guide

This guide explains the features and functions of a Point of Sale (POS) Cart screen, written for a non-technical audience like retail store associates. It focuses on clear, step-by-step instructions for completing daily tasks.

Introduction

The Cart operates as the POS application's idle screen. You can access it from each of the following locations:

  • After logging into the POS UI, you will land on the Cart screen by default.
  • After completing a transaction, you will be automatically redirected to an empty Cart screen.
  • On the upper-left side of the POS UI, expand the main menu and click Cart.

At a high level, the POS Cart is separated into two sections:

  • Item cards: Contains information related to each transaction item, e.g., the Image, SKU, Quantity, Unit Price, and Total Price.
  • Checkout Summary panel: Contains transaction-level information, e.g., the Subtotal, applied Promotion and Discount values, Taxes, and Transaction Total.

Modifying Items in the Cart

Action Description
Add Manual Discount Apply a percentage or fixed amount discount to a specific item. This requires selecting a reason code (e.g., "Damaged," "Staff Discount").
Override Item Price Manually change the unit price of an item. This requires a reason code (e.g., "Product Imported With Incorrect Price").
Modify Item Quantity Increase or decrease the quantity by scanning the item multiple times or using the `+` and `-` buttons on the item card.
Void Item Remove an item from the Cart.

Modifying the Entire Cart

Action Description
Void Cart Removes all items from the transaction.
Suspend Pauses a transaction and prints a receipt with a barcode that can be scanned later to resume the transaction.
Sell Gift Card Activates and loads a specified amount onto a gift card.
Tax Exempt Applies tax-exempt status to the transaction by entering a valid Tax ID and Organization Name.

Change Notice Process Definition

This document defines the internal process, criteria, and scope for creating and delivering customer-facing Change Notices. This ensures clear and timely communication about product updates for stakeholders and is crucial for maintaining trust and alignment.

Criteria and Scope

Change Notices are brief, release-specific documents that contain timely use case instructions, best practices, and/or implementation requirements for specified customer audiences.

While release notes describe the highest level changes to the product, Change Notices typically go into greater detail on how and why a given change impacts a particular user type.

Template Structure

Change Notices are broken into the following major sections:

  • Who does this impact? Call out which customer types are impacted by the change.
  • Effective date: Provide the version number containing the change.
  • What is changing? Describe the customer-facing elements of the application that changed.
  • Why is the change being made? Provide brief context around why the update was implemented.
  • What impact will it have on customers? Describe what the impacted customer type must do as a result of the change.

Change Notice Workflow

  1. 1
    Determine Need

    Product Owner identifies if a feature or enhancement requires a formal Change Notice during sprint planning.

  2. 2
    Draft Content

    PO works with relevant SMEs (developers, designers) to draft the notice and alerts the Documentation Team.

  3. 3
    Edit & Format

    The draft is handed off to the Documentation Team for final editing, formatting, and quality assurance.

  4. 4
    Deploy

    The finalized notice is published to the customer-facing documentation portal and distributed to impacted users.
